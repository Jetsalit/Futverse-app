# FutVerse Firestore Database Design

This document outlines the proposed Firestore database architecture for the FutVerse application, tailored to support its comprehensive suite of features for elite football academies.

## Naming Conventions
- **Collections & Subcollections:** Plural, `snake_case` (e.g., `users`, `scouting_reports`, `fitness_tests`).
- **Fields:** `camelCase` (e.g., `firstName`, `createdAt`, `teamId`).
- **Timestamps:** Standard Firebase `Timestamp` objects (typically suffixed with `At`, e.g., `createdAt`, `updatedAt`).
- **Document IDs:** 
  - `users`: Firebase Auth UID.
  - Other collections: Auto-generated 20-character Firestore IDs.

---

## 1. Collections & Document Schemas

### `users`
Stores application user accounts and global access control data.
- **Document ID:** Firebase Auth UID
- **Fields:**
  - `email` (string)
  - `name` (string)
  - `role` (string) - Enum: `SUPERADMIN`, `ADMIN`, `DATA_ADMIN`, `COACH`, `SCOUT`, `PLAYER`, `USER`, `PARENT`
  - `status` (string) - Enum: `ACTIVE`, `PENDING`, `INACTIVE`
  - `subscriptionPlan` (string, optional)
  - `createdAt` (timestamp)
  - `updatedAt` (timestamp)

### `teams`
Represents different squads within the academy or club (e.g., U18s, First Team).
- **Document ID:** Auto-generated
- **Fields:**
  - `name` (string) - e.g., "U18 Academy"
  - `category` (string) - e.g., "Youth", "Pro"
  - `headCoachId` (string) - Reference to `users` (Coach)
  - `season` (string) - e.g., "2024/2025"
  - `createdAt` (timestamp)

### `players`
Detailed profile information for youth and pro players.
- **Document ID:** Auto-generated
- **Fields:**
  - `userId` (string) - Reference to `users` (Player), optional if profile is managed entirely by staff
  - `teamId` (string) - Reference to `teams`
  - `firstName` (string)
  - `lastName` (string)
  - `position` (string)
  - `dateOfBirth` (timestamp)
  - `nationality` (string)
  - `height` (number)
  - `weight` (number)
  - `isPro` (boolean)
  - `createdAt` (timestamp)

#### Subcollections under `players/{playerId}`:

- **`fitness_tests`**
  - `date` (timestamp)
  - `testType` (string) - e.g., "Yo-Yo Intermittent", "Sprint 30m"
  - `score` (number/string)
  - `evaluatorId` (string) - Reference to `users`

- **`recovery_logs`**
  - `date` (timestamp)
  - `sleepHours` (number)
  - `muscleSoreness` (number) - Scale 1-10
  - `fatigueLevel` (number) - Scale 1-10
  - `notes` (string)

- **`idp_goals` (Individual Development Plan)**
  - `title` (string)
  - `description` (string)
  - `targetDate` (timestamp)
  - `status` (string) - Enum: `TODO`, `IN_PROGRESS`, `COMPLETED`
  - `assignedBy` (string) - Reference to `users` (Coach)

- **`peer_votes`**
  - `matchId` (string) - Reference to `matches`
  - `voterId` (string) - Reference to `players`
  - `category` (string) - e.g., "Man of the Match", "Best Work Ethic"
  - `createdAt` (timestamp)

### `matches`
Information regarding fixtures and match-day setups.
- **Document ID:** Auto-generated
- **Fields:**
  - `teamId` (string) - Reference to `teams`
  - `opponent` (string)
  - `date` (timestamp)
  - `venue` (string) - Enum: `HOME`, `AWAY`, `NEUTRAL`
  - `competition` (string)
  - `result` (string, optional) - e.g., "W 2-1"

#### Subcollections under `matches/{matchId}`:
- **`evaluations`** (Post-match stats & ratings)
  - `playerId` (string)
  - `rating` (number)
  - `minutesPlayed` (number)
  - `goals` (number)
  - `assists` (number)
  - `coachNotes` (string)

- **`starting_xi`**
  - `formation` (string) - e.g., "4-3-3"
  - `lineup` (array of objects) - `[{ position: "ST", playerId: "id123" }]`
  - `tacticalNotes` (string)

### `drills`
Centralized library for training exercises.
- **Document ID:** Auto-generated
- **Fields:**
  - `title` (string)
  - `category` (string) - e.g., "Attacking", "Defending", "Physical"
  - `difficulty` (string) - e.g., "Beginner", "Advanced"
  - `durationMinutes` (number)
  - `description` (string)
  - `diagramUrl` (string)
  - `authorId` (string) - Reference to `users`
  - `createdAt` (timestamp)

### `scouting_reports`
Reports generated by the scouting department on external targets.
- **Document ID:** Auto-generated
- **Fields:**
  - `targetName` (string)
  - `targetTeam` (string)
  - `targetPosition` (string)
  - `scoutId` (string) - Reference to `users` (Scout)
  - `matchObserved` (string)
  - `dateObserved` (timestamp)
  - `overallRating` (number) - Scale 1-10
  - `technicalAssessment` (string)
  - `physicalAssessment` (string)
  - `recommendation` (string) - Enum: `SIGN`, `MONITOR`, `REJECT`
  - `createdAt` (timestamp)

### `notifications`
System alerts and messages for users.
- **Document ID:** Auto-generated
- **Fields:**
  - `userId` (string) - Recipient reference
  - `title` (string)
  - `message` (string)
  - `type` (string) - e.g., "INFO", "ALERT", "TASK"
  - `isRead` (boolean)
  - `createdAt` (timestamp)

## 2. Relationships & Data Flow

- **1-to-1 / 1-to-Many (Users to Players):** A `user` document (with `role: PLAYER`) maps to a `players` profile via the `userId` field. Parents might have an array of `childPlayerIds`.
- **1-to-Many (Teams to Players):** A `team` logically groups players. Queries fetch players where `teamId == targetTeamId`.
- **Many-to-Many (Matches to Players):** Match evaluations connect players to matches via subcollections, allowing efficient querying for a player's historical match performance or a team's match summary.
- **Top-Level vs. Subcollections:** Deeply nested, high-frequency data bound to a specific entity (like `fitness_tests` for a player) is stored in subcollections. Global entities that need cross-referencing (like `drills` or `scouting_reports`) are kept at the root.

## 3. Required Indexes (firestore.indexes.json)

To support the application's dashboard filtering and analytics, the following composite indexes will be required:

1. **Players by Team & Position:**
   - Collection: `players`
   - Fields: `teamId` (ASC), `position` (ASC)

2. **Scouting Reports by Scout & Date:**
   - Collection: `scouting_reports`
   - Fields: `scoutId` (ASC), `dateObserved` (DESC)

3. **Drills by Category & Difficulty:**
   - Collection: `drills`
   - Fields: `category` (ASC), `difficulty` (ASC)

4. **Notifications by User & Read Status:**
   - Collection: `notifications`
   - Fields: `userId` (ASC), `isRead` (ASC), `createdAt` (DESC)

5. **Matches by Team & Date:**
   - Collection: `matches`
   - Fields: `teamId` (ASC), `date` (DESC)
